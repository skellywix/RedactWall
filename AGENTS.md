# AGENTS.md — operating contract for agents working in RedactWall

Read this before changing anything. It is the agent-memory file: how we work, how to prove a change works, the invariants you may not break, and the mistakes not to repeat. Update it when you learn something (see "Mistake log"). `CLAUDE.md`, if present, may extend this.

## What this project is
RedactWall is a compliance-grade DLP layer that inspects prompts/files headed to AI tools and **warns / requires justification / redacts / blocks** by policy, with a tamper-evident audit log. Three sensors (browser `sensors/browser-extension/`, `sensors/endpoint-agent/`, `sensors/mcp-guard/`) all run the SAME local engine and report to one server. Sold to regulated shops (credit unions; NCUA/GLBA/PCI/HIPAA) — an examiner will inspect this.

## How we work (manager-of-agents)

## Commands
- `npm test` (= `node --test`) — unit/integration.
- `npm run sync-check` — verifies `detection-engine/detect.js` == `sensors/browser-extension/lib/detect.js`. MUST stay green.
- `npm run sync-engine` — propagate the shared engine to the copy (the ONLY way to update the copy).
- `npm run train-semantic` — regenerate the on-device semantic model in `detection-engine/detect.js`. Deterministic; CI fails on drift.
- `npm run hooks:install` — install local review hooks into `.githooks`.
- `npm run review:agent` — run the pre-commit review alias locally.
- `npm run review:ci` — full local gate (`git diff --check`, generated demo-doc drift check, AI-domain coverage check, `npm run native:check`, `npm run console:build`, `npm test`, Playwright browser tests, `npm run sync-check`, `npm run eval`).
- `npm run eval` — precision/recall/F1 on the HELD-OUT labeled corpus (`test/fixtures/semantic-eval.json`, 500+ decontaminated cases). Floors: semantic precision >= 0.95, recall >= 0.80, structured recall >= 0.95; enforced by `test/eval.test.js`, composition by `test/eval-corpus.test.js`; **zero benign false positives** is the hard gate.
- `npm run simulate` — end-to-end detection over the sample corpus.
- `npm start` — run the server. `docker build -t redactwall .` — CI also builds the image.

## Change-control process
- Run `npm run hooks:install` after cloning or reinstalling the repo.
- `pre-commit` runs `npm run review:agent` and aborts the commit if checks fail.
- `post-commit` runs `npm run review:ci` and only pushes to `origin` when checks pass.
- If review or push fails, the commit remains local and you can run `git push` after resolving the issue.

## Hard invariants (do not break)
1. Detector logic lives in `detection-engine/detect.js` ONLY. After editing, run `npm run sync-engine`; never hand-edit `sensors/browser-extension/lib/detect.js`.
2. Never hand-edit the semantic model block — regenerate via `npm run train-semantic` (deterministic).
3. `alwaysBlock` types always block/tokenize regardless of mode. Do not weaken. `server/policy.js` (`DEFAULT_POLICY.alwaysBlock`) is the source of truth for the full list — currently 21 types: US_SSN, CREDIT_CARD, BANK_ACCOUNT, ROUTING_NUMBER, IBAN, US_PASSPORT, US_ITIN, US_NPI, MEMBER_ID, LOAN_NUMBER, MEDICAL_RECORD_NUMBER, HEALTH_INSURANCE_ID, UK_NINO, UK_NHS_NUMBER, CANADA_SIN, AUSTRALIA_TFN, INDIA_AADHAAR, SECRET_KEY, PRIVATE_KEY, CANARY_TOKEN, and EXACT_MATCH.
4. No raw PII / prompt text in logs, errors, or the audit `entry` — redacted detections + hashes only.
5. The audit log is append-only and hash-chained: after any `server/db.js` / `server/crypto.js` change, `node -e "require('./server/db').verifyAuditChain()"` must report `ok:true`.
6. Detection changes must keep `npm run eval` floors green — held-out precision/recall AND **zero false positives on benign prompts**. Don't tune the model against `test/fixtures/semantic-eval.json`; it's the held-out test, not training data.

## How to exercise RedactWall end-to-end (required evidence before handing back)
Don't trust green unit tests alone. For the surface you changed, produce a **Testing** section with this evidence (synthetic PII only — never real member data):
- **Detection:** `npm run simulate`; show before/after verdicts for the affected prompts; prove `alwaysBlock` types still block.
- **Browser sensor:** load unpacked `sensors/browser-extension/` in a test Chrome profile, paste `123-45-6789` into a chat box → screenshot the block modal and confirm nothing was sent.
- **Policy modes:** flip `config/policy.json`; show warn / require-justification / redact / block behaving correctly; in redact mode confirm only tokens leave the device.
- **Approval flow:** block → held in queue → admin approve → released; capture the audit entries.
- **Audit integrity:** `verifyAuditChain()` → `ok:true`; a tampered row → `ok:false`.

## Review standards (apply before presenting code)
Functions under ~30 lines; no logic duplicated >2x; no dead code/unused imports; names communicate intent; async paths handle errors; keep the hot detection path allocation-light (it runs on every keystroke/paste); no `SELECT *` on hot SQLite paths. Review in a FRESH context, not the session that wrote the code.

## Mistake log (append when you learn something; don't repeat these)
- 2026-07-11: A durable mutation has a real commit point. Once exact target verification and parent durability succeed, later rollback-artifact or lock-release cleanup failures must not turn the committed operation into a reported failure or trigger a duplicate retry. Return committed success, retain only identity-proven recovery artifacts, emit a sanitized cleanup warning, and latch degraded storage into readiness. Preserve falsy thrown callback values and test every caller that couples a file write to database or audit state.
- 2026-07-11: Cleanup must remain bound to the accepted bytes, not merely a familiar pathname or reused inode. Carry BigInt identity plus a bounded-content digest through asynchronous native handoff processing, use exact handle-backed deletion on Windows and guarded quarantine on POSIX, and preserve replacements on processed, rejected, and replayed paths. Mutable license state needs a trusted-parent bootstrap before any target-local lock or staging file, fd-to-path binding, bounded no-follow runtime reads, and an internal trust capability that callers cannot forge.
- 2026-07-11: A guarded Postgres restore must reject hidden superuser authority (`session_user` must equal the direct non-superuser `CREATEDB` owner), create staging non-connectable, revoke public database access before opening its two-slot guard, and keep every fallible cleanup before final enable. Restore/drill cleanup may use only the returned name/OID/owner identity, wait a bounded interval for already-closing sessions without terminating them, and retain the frozen database plus private audit workspace when ownership or cleanup is uncertain.
- 2026-07-11: A Postgres audit scope is database-local authority and must not be copied by `pg_dump`. Exclude the scope table, verify restored rows against the source checkpoint before any scope mutation, mint a fresh target scope, and re-sign only the private staged runtime state before atomic publication. Two restores of one portable backup must receive distinct scopes and reject each other's audit directories. Failed exclusive state publication must quarantine the public path before checking its BigInt identity and preserve any changed replacement.
- 2026-07-11: Publication is not complete when the last rename succeeds. Retain exact prior artifacts until target-layout verification passes, bind every staged/published/rollback path with BigInt identity, quarantine before deletion, and preserve/report changed replacements. A post-publication pending-file check outside rollback scope can otherwise report failure after permanently discarding the valid prior store.
- 2026-07-11: `pg_restore --clean` is not an in-place database replacement: objects absent from the archive survive. PostgreSQL restore must require an absent final name, authenticate the source PG16/17 encoding and locale definition, restore through a cryptorandom owner-only `template0` staging database, freeze and rename the exact OID, publish a new bound runtime audit directory, and enable connections last. A drill owns and removes a database only through the exact identity returned by a successful guarded restore; uncertain outcomes stay frozen for manual recovery.
- 2026-07-11: A documented container backup path must ship compatible `pg_dump` and `pg_restore` binaries. Prove their versions during the image build and from the final non-root image, forward the required audit directory/key through Compose, and require a client major at least as new as the server.
- 2026-07-11: Windows `fs.stat` Number identities are not exact: distinct NTFS file IDs above `Number.MAX_SAFE_INTEGER` can compare equal and let path-to-handle swaps pass. Every authoritative file, directory, lock, staging, license, endpoint, and native-handoff identity proof must use `{ bigint: true }`, require nonzero device/file IDs and the expected link/type contract, and compare nanosecond metadata where stability matters. Keep a deterministic rounded-ID collision regression.
- 2026-07-11: Migration 8 may become durable only after the exact pre-v8 audit tail has verified and its authenticated checkpoint is durably published under the initialization lock. A fresh protocol-v2 state rejects legitimate legacy rows, while a reusable v1 state without a migration-bound authorization lets a crash-window attacker inject an un-MACed tail. The pre-migration hook must create the checkpoint first, and a post-migration restart without it must fail closed.
- 2026-07-11: Multi-host Postgres needs one coherent durable audit-anchor volume shared by every replica and Postgres advisory coordination for all startup, pending, checkpoint, reconciliation, and readiness operations. Replica-local or copied sidecars fail after another host commits, while hostname/PID file locks cannot safely recover a crashed remote owner. Keep SQLite on filesystem coordination, require an explicit absolute shared audit directory for Postgres, and prove independent directories fail closed.
- 2026-07-11: A restore is incomplete until it publishes the authenticated runtime audit state that the restored server will actually open. SQLite needs a newly HMAC-authenticated target-layout manifest; Postgres needs a new explicit runtime audit directory containing the exact state/checkpoint and no pending witness. Prove the documented verification command and a clean `server/db` boot, and never delete a cleanup target whose exact created identity is no longer proven.
- 2026-07-11: Durable Windows security work makes fixed sleeps invalid release evidence. Await the returned job or a named user-visible outcome before asserting logs or persisted rows, and drain every concurrent child before cleanup so one rejection cannot race teardown. Browser fixtures must bind their native listener to the exact send control and prove coaching plus submission under the same policy state. Give production SQLite's measured multi-process private-directory first boot an explicit 60-second timeout and maximum, while routine mutation locks remain capped at 30 seconds.
- 2026-07-11: Windows audit commits cannot afford repeated full owner/ACL subprocesses, but the fast path must carry the complete startup proof. Keep bootstrap writes on full owner/reset/grant hardening; pin every sidecar directory to nonzero device, file id, and creation identity; bracket trusted bounded reads and removals; re-authenticate durable state without permitting an embedded-key rotation; and keep every fallible final pending-publication check inside the exact-prior rollback scope. NTFS preserves the replaced destination creation time, so bind atomic file replacement by nonzero device/file id, size, and single-link state instead. Ordinary lock contention is transient, while storage, trust, or cleanup failures must freeze readiness.
- 2026-07-10: Never use an external checkpoint's count, sequence, or head before its MAC verifies. In protocol-v2 audit recovery, authenticate any present checkpoint immediately after the explicit missing-checkpoint check and before pending-gap, truncation, tail, or fast-path arithmetic; otherwise a forged count can steer failure classification and later logic even when the system ultimately fails closed.
- 2026-07-10: An authenticated SQLite restore is not production-ready if its database is published before the destination parent becomes trusted private state. Establish the missing or empty parent under the shared private-directory lock before creating staging or rollback artifacts, hold that lock through publication, and reject pre-existing state in an untrusted directory. Prove the restored database can open under production fail-closed storage checks on both Windows and Linux.
- 2026-07-10: Windows owner and ACL verification can make healthy multi-process first boot exceed routine mutation budgets. Give session-secret and other measured initialization-only paths an explicit 60-second timeout and explicit 60-second maximum while keeping routine locks capped at 30 seconds. First-boot tests must isolate every data path under their temporary root, assert final file privacy rather than implementation call counts, and use a bounded Windows readiness deadline that covers required OS security probes.
- 2026-07-10: An authenticated audit row beyond an older checkpoint is not durable truncation evidence: a crash before deferred checkpoint publication followed by tail deletion can make restart accept the old checkpoint. Before the database can commit, durably publish an independently HMAC-authenticated pending high-water bound to the exact entry id, sequence, count, and head; hold its interprocess lock through the transaction outcome, keep the prior pending state when the callback rolls back before preparation, and reconcile before startup readiness. Any error after preparation is commit-uncertain because COMMIT may have applied while its response was lost: retain the new pending record and fail health until the exact tail is proved, never restore it without conclusive rollback evidence. Merge only released nested savepoints, use `BEGIN IMMEDIATE` plus a bounded busy timeout for cross-process SQLite head reads, and retain the Postgres transaction advisory lock.
- 2026-07-10: Authenticating a PostgreSQL backup pathname does not bind the bytes later opened by `pg_restore`. After verification, copy the dump and required sidecars through stable no-follow handles into a private local staging directory, enforce the authenticated byte counts and sidecar ceilings, fsync the snapshot, then rehash and authenticate that staged set. Invoke `pg_restore` only on the staged dump and retain cleanup through post-restore verification and every failure path.
- 2026-07-10: A database audit row can commit before its deferred checkpoint publication fails. Retain that exact authenticated tail as an in-memory high-water mark, fail readiness, and freeze later audit-coupled mutations until synchronous repair both proves the tail still exists and durably advances the checkpoint. File-backed license installation must hold one interprocess lock across the fresh byte snapshot, publication, audit commit, exact-byte rollback, and in-memory refresh so one replica cannot stale-rollback another replica's successful install.
- 2026-07-10: Every Windows directory that can contain authoritative local state needs the trusted-profile bootstrap before its first file exists. Endpoint handoff and consumed-claim spools, gateway bearer-token stores, packaged runtime configuration, and production SQLite directories must reject any entry found under a broad DACL, establish trust before first publication, and never repair already-present state into acceptance; SQLite keeps the bootstrap lock through database open. Exercise each boundary with planted state plus concurrent fresh-process and crashed-lock recovery tests, and make first-boot seeders establish the shared data-directory trust before the server opens storage.
- 2026-07-10: Backup artifact hashes are not authentic when an attacker can rewrite both the database and its manifest, and a MAC key embedded beside the backup is attacker-recoverable. Require an externally configured audit key, HMAC the complete canonical SQLite and PostgreSQL manifest, verify the MAC in constant time before restore, reject embedded-key, unsigned, or missing manifests, and keep `--force` limited to replacing an existing target rather than bypassing backup authentication.
- 2026-07-10: Chromium writes a root `_metadata/` cache beside unpacked extension sources. Package and asset-budget walkers must exclude that exact browser-owned root only, while authored nested files remain counted. Consolidate repeated security flows before recalibrating feature-driven ceilings, retain measured headroom instead of hiding growth with broad exclusions or unreadable minification, and preserve urgent coaching precedence across the complete finding set because a lower-risk email or phone match may appear before a credential. - 2026-07-10: Shared Windows audit-sidecar directories must establish trust under the user-profile bootstrap lock before taking the checkpoint lock, and the bootstrap lock must span state creation plus first-boot migrations so directory identity is rechecked before readiness. Reject every preexisting entry in an untrusted directory, reclaim only conclusively dead bootstrap owners, and make later sidecar writes assert the directory ACL instead of racing replicas through repeated `icacls` resets.
- 2026-07-10: Deployable readiness must prove the configured credential on a bounded, non-writing authenticated endpoint as well as public dependency health. Send the credential only to the exact configured HTTPS or loopback origin, reject redirects, and never let a development cleartext override authorize credential transport to a remote host.

Serialize each customer through a transactional high-water CAS, commit its audit evidence in the same transaction, treat authenticated audit history as authoritative, reconcile before every authorization and status surface, and drain binaries that predate the shared-state migration before restoring traffic.

HMAC the complete manifest that binds the dump and every sidecar to the exact checkpoint and database snapshot, reject missing or rewritten manifest authentication even under force, and require a fresh backup when upgrading from an unsigned format.
