# Release Process

How RedactWall versions, builds, and ships releases. Modeled on the release
practices of GitLab (self-managed release trains, backport policy) and
HashiCorp (signed artifacts, checksums), scaled to a small team.

## Versioning

RedactWall follows [Semantic Versioning 2.0.0](https://semver.org/spec/v2.0.0.html):

- **MAJOR** — breaking change: config schema, HTTP API, sensor policy
  contract, or an upgrade that requires operator action beyond
  restart-and-migrate.
- **MINOR** — new features, new detectors, new console surfaces.
- **PATCH** — bug fixes and security fixes only.

The `detection-engine/` public API is additive-only inside a major version —
all sensors depend on it.

## Cadence

- **Minor releases ship on a monthly train** (first week of the month). A
  feature that misses the train waits for the next one; the train does not
  wait for features.
- **Patch releases ship as needed**, typically within days for Sev1/Sev2
  fixes (see `SECURITY.md` and `docs/process/SUPPORT_POLICY.md`).
- **Security backports** go to the current and previous minor release. State
  this to customers so they can plan upgrade windows — regulated buyers cite
  this policy in vendor reviews.

## Release rings (the on-prem substitute for canary)

1. **Ring 0 — our own instance.** The release candidate runs on the internal
   demo/staging instance for at least a week before tagging.
2. **Ring 1 — early-access customers.** One or two friendly pilot customers
   get the release first, with a named contact and a fast patch path.
3. **Ring 2 — general availability.** Announce via release notes; update the
   `Updates` tab feed (the console links to GitHub Releases).

Preflight is the safety net at every ring: `npm run setup:check` and
`GET /readyz` validate Node version, secrets, SQLite path safety, audit-chain
integrity, and config schema before and after upgrade.

## Cutting a release

1. Verify the gates on the release commit:
   ```bash
   npm run review:ci      # whitespace, generated docs, AI domains, tests, browser E2E, engine sync, eval
   npm run suite          # standalone black-box regression suite (see docs/process/TESTING_STRATEGY.md)
   node -e "const v=require('./server/db').verifyAuditChain(); if(!v.ok) process.exit(1)"
   ```
2. Move the `[Unreleased]` section of `CHANGELOG.md` under the new version
   heading with today's date; keep `Added / Changed / Fixed / Security /
   Deprecated / Removed` subsections. Add the version compare link.
3. Bump `version` in `package.json` to match.
4. Tag `vX.Y.Z` and push the tag. Create the GitHub Release with the notes
   copied from the changelog section.
5. Build and attach artifacts:
   ```bash
   npm run package:extension        # browser extension zip + SHA-256 manifest
   npm run package:endpoint-agent   # endpoint agent zip + SHA-256 manifest
   npm run package:mcp-guard        # MCP guard zip + SHA-256 manifest
   npm run security:package:zip     # trust package with CycloneDX-style SBOM
   docker build -t redactwall:vX.Y.Z .
   ```
   Attach the zips, their SHA-256 manifests, and the SBOM to the release.
6. Announce to Ring 1, then Ring 2.

Every packaging script already refuses development ingest keys and emits
hash manifests; do not ship artifacts that were not produced by these scripts.

## Supply-chain posture (SLSA direction)

- Builds should move to a GitHub Actions release workflow triggered by the
  tag, so provenance is generated by hosted CI rather than a laptop
  (SLSA level 2 direction). Keyless `cosign` signing of the Docker image and
  zips is the target once the workflow exists.
- Until then: artifacts are built from a clean checkout at the tag, and the
  SHA-256 manifests published with the release are the integrity reference.

## Upgrade guidance for operators

- Read the release notes for the versions being skipped.
- Back up first: `npm run backup` (verifier-first manifest).
- Upgrade, then confirm `GET /readyz` and audit-chain integrity.
- Rollback = restore the previous version and `npm run backup:restore`.
  Patch releases never change the database schema; minor releases that do
  include a migration note in the changelog.
